Does XQuartz launch whenever you fire up vim, causing slow startup? Turns out vim feels out its environment on startup by checking variables, including $DISPLAY, whose value is something like 

        
          /private/tmp/com.apple.launchd.muoEIdGNPd/org.macosforge.xquartz:0
Somehow that check causes XQuartz to launch (even though vim won't use it). My solution is to set $DISPLAY to null only for vim-related commands by aliasing them in ~/.bash_profile (which Mac uses instead of ~/.bashrc): 
        
          # prevent xquartz from opening
          alias view='DISPLAY="" view'
          alias vi='DISPLAY="" vi'
          alias vim='DISPLAY="" vim'
          alias vimdiff='DISPLAY="" vimdiff'
          alias vimtutor='DISPLAY="" vimtutor'
You can take a similar approach to avoid the 
        
          crontab: temp file must be edited in place
error when editing your crontab with vim. Just add: 
        
          alias crontab="VIM_CRONTAB=true crontab"
to your bash dotfile, then edit your .vimrc file to include: 
        
          :if $VIM_CRONTAB == "true"
          :set nobackup
          :set nowritebackup
          :endif

Useful LIGO Links

Getting started with LIGO is somewhat difficult due to the scattered nature of documentation. Below are some useful links that are very easy to forget. They all require LIGO credentials.

  • I maintain a virtual machine with all the LIGO software you probably need installed on it; you should probably use this virtual machine for almost any work you are going to do (particularly if you are with the Columbia gravitational wave group).
  • If you need to install LIGO software, here is a collection of links to installation instructions for different operating systems , including Debian 8, SL7, and Mac OS.
  • Need LIGO authentication for a command-line script, like a curl request to GraceDB? Go here for robot certificate signup information. The form is available here.
  • A great introduction to data flow in LIGO on the LigoDV-Web site (along with help on using LigoDV-Web) can be found here . LigoDV-Web is a good tool for generating quick plots of small chunks of data using a web-GUI (which also makes is usable from any device). It even prints the plotting command used to generate each plot. That said, it relies on NDS and is no good for anything involving large quantities of data.
  • This is how you parse LIGO/VIRGO GCN Notices for EM Followup .
  • Here is a list of Gravitational Wave Frame Data Types .
  • Go to my.ligo.org to manage basic account information or to sign up for LIGO credentials.
  • Wondering when the next Engineering Run or Observation Run is going to happen? Look up dates for events on the aLIGO Comissioning Calendar .
  • You can manage your email listserv subscriptions at sympa.ligo.org or grouper.
  • LIGO has way too many acronyms.
  • The aLIGO Wiki Timing Page , specifically Timing FPGA code. There is a lot of stuff on the rest of the wiki that is useful.
  • The CDS (Control and Data System) pages at the two sites: LLO CDS and LHO CDS. These are a great starting point for any work you need to do, especially on site; they contain up-to-date useful links.
  • If you want to log into LIGO servers (which is something you almost certainly will want to do), follow these instructions to get LIGO DataGrid credentials.
  • This guy has some good tips, but they're not all good, so make sure to think for yourself.
  • If you are trying to remotely access data, e.g. get timeseries or view MEDM controll screens from offsite, you should visit this wiki page.
  • If you need detailed logs of goings on at the observatories, visit the Hanford alog site for logs regarding the Hanford detector and the Livingston alog site for logs regarding the Livingston detector.
  • Every file you could ever need is ostensibly stored on DCC, though finding the file you need will likely be impossible if you don't already know how to search for it. (Sorry!)
  • LIGO's subversion repositories, containing much important code, are here. If you ever want to work on a subversion repository, you can "checkout" the repository (this is equivalent to git clone url) with svn checkout --username albert.einstein url, where, of course, the username is your own and the URL is that of the repository you want to work with. Be prepared for a very big disk footprint and a very long wait; some of these repositories are huge. If you want a more in-depth description of SVN (tailored to Git users), check out Git - SVN Crash Course .
  • More general links about CDS.

It is also worth knowing that there is a new type of frame in addition to the standard ones, which I believe is dedicated to h(t): L1_HOFT_C01 and H1_HOFT_C01.

Controversial Opinions, or, an Invective Against Pythonistas

bash has idiosyncracies but, unlike python and other hyped languages of the 2000s, it has poetry. Many people are quite confused by this and mistake e.g. their own fear of regular expressions for a sign that sed is hard to use. The UNIX philosophy is similarly all-right with me. Simple but powerful is good.

Many people think they need IDEs; this is false. The command line is an IDE. vi, sed, grep et al. are the best editing tools around. And the command line is extensible.

At the other end of the self-consistency spectrum is julia, whose creators' relentless persuit of crystalline conceptual perfection has wrought a terse, fast, clean, and—like bash— flexible language. In julia, everything makes sense. Everything is beautiful. There is more than one way to do things because we are all adults.

I have not learned perl, but the fact that "pythonistas" seem to dislike it suggests that it is a very good language.

python is a good language. For many tasks it is the most reasonable choice. Trying to argue that it is always the best scripting language to use, or that other scripting languages are foul-smelling, is somewhat like insisting that you are only capable of eating chicken nuggets.

Compare The Tao of Programming:

"Technique?" said the programmer, turning from his terminal, "What I follow is Tao -- beyond all techniques! When I first began to program, I would see before me the whole problem in one mass. After three years, I no longer saw this mass. Instead, I used subroutines. But now I see nothing. My whole being exists in a formless void. My senses are idle. My spirit, free to work without a plan, follows its own instinct. In short, my program writes itself. True, sometimes there are difficult problems. I see them coming, I slow down, I watch silently. Then I change a single line of code and the difficulties vanish like puffs of idle smoke. I then compile the program. I sit still and let the joy of the work fill my being. I close my eyes for a moment and then log off."

to The Zen of Python:

There should be one-- and preferably only one --obvious way to do it.

What a sad way to see the world!

Disclaimer: I enjoy python itself. I am only complaining about python-novice fanboyism.
